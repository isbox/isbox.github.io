强制类型转换

显式的将值从一种类型转为另外一种类型称为类型转换，如果是隐式的情况则被称为强制类型转换

另一种说法：类型转换发生在静态语言类型的编译阶段，而强制类型转换则是在动态类型语言运行时（runtime）

Javascript中都统称为强制类型转换，如果要做区分，分为显式强制类型转换与隐式强制类型转换

JavaScript 中的强制类型转换总是返回标量基本类型值，如 字符串、数字、布尔值，不会返回对象和函数

⚠️封装并非严格意义 上的强制类型转换

抽象值操作（仅供内部使用的操作？）

首先，我们要了解下，字符串、数字、布尔值间的转换规则

ToString

toString 定义了非字符串到字符串的强制类型转换；

对于基本类型值

对于普通对象，如果有自己的 toString 方法，字符串化时就会调用该方法并 使用其返回值；

反之，会调用 toString（Object.prototype.toString） 返回内部属性 [[Class]] 的值，如 [object Object]；

数组拥有自己的 toString 方法，会将所有单元字符串化后使用 ‘，’ 连接

JSON 字符串化

工具函数 JSON.stringify 在将 JSON 对象序列化为字符串时也用到了 ToString

JSON 字符串化并非严格意义上的强制类型转换，因为其中也涉及 ToString 的相关规则

对于大多数简单值来说，json字符串化效果与 toString() 相同

所有安全的 JSON 值(JSON-safe)都可以使用 JSON.stringify(..) 字符串化

安全的JSON值是指能够呈现为有效 JSON 格式的值

ToBoolean

Javascript 中有两个关键词 true / false，分别代表布尔类型的真 / 假；

我们经常将数字 1 / 0 去对应 true / false 实际上它们并不是一回事，只是Javascript可以将 1 强制转换为 true，0 强制转换为 false；

Javascript中的值分两类

  • 可以被强制类型转换为 false 的值
  • 其他（可以被强制转换为 true 的值）

ECMA规范规定了 toBoolean 的转化规则，见 ECMA ToBoolean

假值对象

规范上面说了对象都会反映为真值，为何还有假值对象；

new Boolean(false) ? new String('') ? 这样的算假值对象？

其实上面两个并非假值对象

假值对象实际并不属于Javascript语言范畴 ，浏览器在某些特定情况下，在常规JavaScript语法基础上自己创建了一些外来(exotic) 值，这些就是假值对象

假值对象看起来与Javascript普通对象没有区别，但是它们强制类型转换为 boolean 时为 false；

常见例子为 document.all，它为一个数组对象，包含了页面上的所有元素，由DOM(而不是 JavaScript 引擎)提供给 JavaScript 程序使用

document.all 在之前为一个真正的对象，且强制转换布尔类型为 true，但 document.all 并非标准，随着浏览器标准推进而被废弃了，但是浏览器无法完全将它去除，于是将它以假值处理

+ 为一元操作符，运算符会显示转为数字，而不是将其带入计算再转；

~符号

~ 运算符（字位操作“非”），字位运算符只适用于 32 位整数（通过抽象操作 ToInt32 来实现，ToInt32 会调用 ToNumber 强制类型转换）

由于 ~ 只能用于 number 类型，返回值类型没发生变化，不存在被强制转换类型，但是某些特殊数字会产生转换效果

~ 首先将值强制类型转换为 32 位数字，然后执行字位操作“非”（对每一个字位进行反转）

-1 是一个“哨位值”，哨位值是那些在各个类型中(这里是数字)被赋予了特殊含义的值。 在 C语言中我们用 -1 来代表函数执行失败，用大于等于 0 的值来代表函数执行成功

~ 的一个合适的使用方式在于对 indexOf 未匹配到的情况专为 0 值，可以做 boolean 判断

显式解析数字字符串

显示解析字符串与强制类型转换为数字，虽然返回的结果都是数字，但是解析和转换还是有所区别的

解析允许字符串中还有非数字字符，解析会按照从左到右的顺序进行，遇到非数字字符停止

转换不允许还有任何非数字字符，否则返回 NaN

parseInt，处理非字符类型的参数时会先强制转换为字符串，应该避免这样做，会产生奇怪的问题

ES5之前 parseInt 存在一个坑，回根据第一个字符自动选择进制

- 如果第一位是 `x` -> 转为十六进制

虽然ES5之后默认为十进制，但是 js 书写规范中仍然后一条 parseInt 必须传入第二个参数

宽松相等（==）和 严格相等（===）

== 和 === 都是用来判断两个值是否 “相等”，但是它们之间有一些区别，尤其是在判断条件上；

:x: 有一个误区是：== 检查值是否相等，=== 检查值和类型是否相等

:white_check_mark: 正确的解释是：== 允许在相等比较中强制类型转换，而 === 不允许

对于以上错误的解释，=== 会多一步骤，二正确的解释 == 会多一个步骤，但是不论何种，对于性能的影响微乎其微；

== 和 === 都会检查操作数的类型，区别在对其处理方式不同；
